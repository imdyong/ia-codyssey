>
> 한 번쯤 이런 허탈함을 느껴본 적이 있을 겁니다.  
> 분명 세상을 바꿀 혁명이라고 하는데, 막상 내 업무를 맡기면 엉뚱한 대답만 늘어놓거나 겉도는 말만 반복하는 AI.  
> 그런데 옆자리 동료는 같은 AI로 보고서 초안부터 데이터 요약까지 뚝딱 해치웁니다.  
> **차이는 바로 질문(프롬프트) 설계에 있습니다.**

## 1. 🎯 미션 소개
같은 모델이라도 **프롬프트 한 줄**이 달라지면 결과물의 수준은 완전히 바뀝니다.  
이번 미션은 '그냥 물어보는 것'과 '제대로 시키는 것' 사이의 간극을 직접 체감하는 것이 출발점입니다.

### 단계별 활동
| 단계 | 활동 |
|------|------|
| 1️⃣ | 3종 이상의 LLM을 동일한 업무 과업에 투입해 성능 비교 → "느낌"이 아닌 평가 축과 근거로 최적 모델 선정 |
| 2️⃣ | 특정 페르소나를 가진 AI 챗봇의 시스템 프롬프트 직접 설계 |
| 3️⃣ | Few-shot 예시와 단계적 추론 유도 기법을 적용해 프롬프트를 버전별로 개선 |
| 4️⃣ | 10턴 이상의 대화에서 문맥을 유지하며 논리적으로 대화를 이끌어가는 텍스트 생성 파이프라인 구축 |
| 5️⃣ | **환각(Hallucination)**을 체계적으로 검증 |

- 📄 **시스템 설계 문서**

## 3. 🏁 과제 목표
이 과제를 마친 후 학습자는 아래를 설명할 수 있어야 합니다:

- [ ] LLM이 "확률적 생성" 기반으로 동작하며, 환각(Hallucination)이 발생하는 이유  
- [ ] Zero-shot / Few-shot / Reasoning Prompt 기법의 차이와 적용 방법  
- [ ] 3종 이상의 LLM을 비교할 때, 평가 축과 근거를 통해 선택 이유 설명  
- [ ] 페르소나와 시스템 프롬프트가 결과 품질에 미치는 영향  
- [ ] 생성 결과를 테스트로 검증하고, 프롬프트를 반복 개선하는 과정 재현

## 4. ✅ 기능 요구 사항

### 4-1. 모델 비교 및 선정
- 최소 3종 LLM 투입 (예: GPT-4o, Claude 3.5 Sonnet, Gemini 1.5 Pro)  
- 보고서 필수 항목: 모델명, 사용 환경, 평가 축(점수표), 최종 선정 근거  

### 4-2. 업무 자동화 과업 정의
- 예시: ✉️ 메일 초안 작성 / 📝 회의록 요약 / 📊 보고서 개요 생성 / 📋 요구사항 정리  
- 문서에 포함: 타겟 사용자, 입력 데이터 형태, 출력 규격, 입력 템플릿  

### 4-3. 페르소나 및 시스템 프롬프트
- 이름/역할/전문 분야/말투/금지 사항/우선순위 정의  
- 시스템 프롬프트 필수 요소: 역할과 목표, 출력 형식 규칙, 안전장치, 사실 처리 규칙  

### 4-4. Few-shot 예시
- 최소 3개 제공 (모호한 입력 케이스 포함)  

### 4-5. 단계적 추론 유도
- v1: 일반 지시  
- v2: 단계적 접근 (요구사항 정리 → 누락 질문 → 초안 작성)  
- 최종 출력: 핵심 근거 3개만 bullet로 제시  

### 4-6. 환각 검증
- 사실 기반 검증 질문 5개 이상 준비  
- ✅ Pass: 정답+근거 OR "확인 필요" + 확인 절차 제안  
- ❌ Fail: 틀린 사실 단정 / 불명확 질문에 임의 답변  

### 4-7. 실행 로그
- 10턴 이상 대화 전문 기록  
- 조건 변경/추가 정보 제공 상황 포함  
- 문맥 유지 판정: 요구사항 일관 유지, 조건 변경 시 해당 부분만 반영

## 6. 🚧 제약 사항
- 모델/환경 기록 필수 (모델명, 요금제, 사용 채널, 사용 날짜, 주요 설정)  
- 무료 버전 사용 시 제한점 1줄 명시  
- 안전/윤리: 폭력·선정적 콘텐츠 금지, 개인정보 포함 금지, 사실 응답 시 근거 또는 "확인 필요" 표기  
- 제출물 편집: 대화 로그는 가독성 있게 정리, 원본 로그 별도 보관

## 7. 💡 결과 예시
> ⚠️ 아래는 참고 예시입니다. 실제 문구와 구성은 달라도 됩니다.

- 📊 **모델 비교 보고서 일부**  
  - 평가 축: 정확성 / 한국어 자연스러움 / 긴 문맥 유지 / 형식 준수 / 속도 / 비용 / 도구 연동  
  - 결론: "회의록 요약" 과업은 긴 문맥 유지와 형식 준수가 중요 → 해당 축 점수가 높은 모델 채택  

- 📝 **업무 과업 입력 템플릿**
```text
[업무 과업] 회의록 요약
[원문] (회의 메모/녹취 요약 텍스트)
[원하는 결과] 결정사항 / Action Items / 리스크 / 후속 일정
[톤] 간결, 사내 공유용
[금지] 추측성 표현 금지, 인물 실명 대신 역할명 사용
[확인 질문 규칙] 정보가 부족하면 최대 3개까지 질문 후 요약 시작
