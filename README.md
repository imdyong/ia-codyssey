> 💬 **"ChatGPT한테 물어봤는데, 생각보다 별로던데요?"**
>
> 한 번쯤 이런 허탈함을 느껴본 적이 있을 겁니다. 분명 세상을 바꿀 혁명이라고 하는데, 막상 내 업무를 맡기면 엉뚱한 대답만 늘어놓거나 겉도는 말만 반복하는 AI. 그런데 옆자리 동료는 같은 AI로 보고서 초안부터 데이터 요약까지 뚝딱 해치웁니다. 도대체 뭐가 다른 걸까요?
>
> **답은 의외로 간단합니다. 질문을 어떻게 설계했느냐, 그 차이입니다.**

---

## 1. 🎯 미션 소개
같은 모델이라도 프롬프트 한 줄이 달라지면 결과물의 수준은 완전히 바뀝니다. '그냥 물어보는 것'과 '제대로 시키는 것' 사이의 간극을 직접 체감하는 것이 이번 미션의 출발점입니다.

이번 미션에서는 대규모 언어 모델(LLM)의 작동 원리를 이해하고, 프롬프트 엔지니어링을 통해 원하는 텍스트 결과물을 정교하게 제어하는 기술을 익힙니다.

| 단계 | 활동 |
| :--- | :--- |
| **1️⃣** | 3종 이상의 LLM 성능 비교 및 "느낌"이 아닌 근거 기반 최적 모델 선정 |
| **2️⃣** | 특정 페르소나를 가진 AI 챗봇 시스템 프롬프트 직접 설계 |
| **3️⃣** | Few-shot 예시와 단계적 추론 유도 기법 적용 및 프롬프트 개선 |
| **4️⃣** | 10턴 이상의 문맥 유지 텍스트 생성 파이프라인 구축 |
| **5️⃣** | **환각(Hallucination)** 체계적 검증 |

이 과정을 마치면 AI를 단순 검색 대용이 아닌, 업무 생산성을 끌어올리는 핵심 파트너로 다룰 수 있게 됩니다.

---

## 2. 📦 최종 결과물
다음 3가지 산출물을 포함한 **업무 자동화 프롬프트 패키지** 1개를 완성합니다.

*   📄 **① LLM 모델 비교·선정 보고서 (문서 1개)**
    *   3종 이상의 모델 입력 테스트 및 평가표, 최종 선정 근거 정리
*   📄 **② 시스템 설계 문서 (문서 1개)**
    *   페르소나, 시스템/유저 프롬프트, Few-shot 예시, v1→v2 개선 이력 및 최종 프롬프트 전문
*   📄 **③ 실행 로그 (대화 로그 1개)**
    *   10턴 이상의 대화 전문 및 문제 발생 지점(환각/문맥 끊김 등) 수정 결과 요약

---

## 3. 🏁 과제 목표
*   [ ] LLM의 "확률적 생성" 및 환각(Hallucination) 발생 원리 설명 가능
*   [ ] Zero-shot / Few-shot / 단계적 추론 기법 구분 및 적용
*   [ ] 데이터 기반(평가 축 활용)의 LLM 모델 비교 선정
*   [ ] 페르소나 및 시스템 프롬프트가 결과 품질에 미치는 영향 분석
*   [ ] 생성 결과의 테스트 검증 및 반복 개선 프로세스 재현

---

## 4. ✅ 기능 요구 사항

### 4-1. 모델 비교 및 선정 (최소 3종)
*   **비교 대상:** GPT-4o, Claude 3.5 Sonnet, Gemini 1.5 Pro 등 (최신 모델 권장)
*   **필수 항목:** 모델명, 사용 환경, 평가 축 4개 이상(모델별 1~5점 점수표), 최종 선정 근거(3줄 이상)

### 4-2. 업무 자동화 과업 선택 및 정의
*   **과업 예시:** 메일 작성, 회의록 요약, 보고서 개요, 요구사항 정리 중 택 1
*   **필수 명시:** 타겟 사용자, 입력 데이터 형태, 출력 규격, **입력 템플릿(재사용 가능 형태)**

### 4-3. 페르소나 및 시스템 프롬프트 설계
*   **페르소나 정의:** 역할/직무/전문분야/말투/금지사항/우선순위
*   **시스템 프롬프트 필수:** 역할과 목표, 출력 형식, 안전장치, 사실/정책/수치 처리 규칙

### 4-4. Few-shot 예시
*   "좋은 입력 → 좋은 출력" 예시 최소 3개
*   ⚠️ **주의:** 3개 중 1개는 반드시 "모호한 입력" 케이스를 포함하여 AI의 되묻기 과정 확인

### 4-5. 단계적 추론 유도 (v1 vs v2)
*   **v1:** 일반 지시 / **v2:** 단계적 접근 유도
*   단, 최종 출력에는 장문의 추론 과정 대신 핵심 요약/근거 위주로 제시 규칙 적용

### 4-6. 환각(Hallucination) 검증
*   사실 기반 검증 질문 5개 이상
*   **Pass 기준:** (A) 정답 및 근거 제시, (B) 모를 경우 확인 절차 제안
*   **Fail 기준:** 근거 없는 단정, 전제 불명확 시 임의 생성

### 4-7. 10턴 이상 대화 로그 + 문맥 유지
*   조건 변경(톤/대상/정책 변경) 또는 추가 정보 제공 시나리오 포함
*   10턴 동안 요구사항 일관성 유지 검증

---

## 5. 🚧 제약 사항 및 제출 가이드
*   **재현성 기록:** 모델명, 요금제, 사용 채널, 날짜, 설정(temp 등) 필수 명시
*   **안전/윤리:** 개인정보 포함 금지, 폭력적/선정적 콘텐츠 생성 금지
*   **사실 검증:** 사실/정책/수치 답변은 "근거/확인 필요" 규칙 준수
*   **편집:** 로그 전문은 별도 파일로 보관, 제출용 로그는 가독성을 위해 요약 및 섹션 정리
