## 1. 미션 소개

> **"ChatGPT한테 물어봤는데, 생각보다 별로던데요?"**

같은 모델을 써도 결과물의 수준이 크게 달라지는 이유는 대개 모델 자체보다 **프롬프트 설계 방식**에 있습니다.  
이번 미션은 LLM을 단순 질의응답 도구가 아니라, **업무 결과물을 안정적으로 생산하는 파이프라인**으로 다루기 위해 필요한 비교·설계·검증 과정을 직접 수행하는 프로젝트입니다.

핵심은 세 가지입니다.

1. **비교** — 3종 이상의 LLM을 동일 과업에 투입해 결과를 평가 축 기준으로 비교한다.  
2. **설계** — 특정 업무에 맞는 페르소나, 시스템 프롬프트, Few-shot 예시, 입력 템플릿을 만든다.  
3. **검증** — 환각과 문맥 이탈을 테스트하고, 프롬프트를 v1→v2로 개선한다.

## 2. 최종 결과물

### 2.1 LLM 모델 비교 · 선정 보고서
동일한 업무 과업 프롬프트를 3종 이상의 모델에 입력하고, 모델별 결과 비교표와 최종 선정 결론을 정리한 문서

### 2.2 시스템 설계 문서
타겟 사용자, 문제 정의, 페르소나, 시스템/유저 프롬프트, Few-shot 예시, 환각 검증 전략, 개선 이력을 담은 문서

### 2.3 실행 로그
10턴 이상 실제 대화 시나리오와 조건 변경, 문맥 유지 여부, 개선 포인트를 정리한 로그

<a id="4-업무-자동화-과업-정의"></a>

## 4. 업무 자동화 과업 정의

### 4.1 선택 과업
**보고서 개요 생성** — 목차 / 핵심 메시지 / 리스크 도출

### 4.2 문제 정의
사용자는 보고서의 목적은 알고 있지만, 초안 단계에서 다음을 빠르게 구조화하는 데 어려움을 겪습니다.

- 어떤 순서로 목차를 잡을지
- 어떤 메시지를 핵심으로 둘지
- 어떤 리스크를 명시해야 할지

### 4.3 타겟 사용자
- 사내 보고서를 자주 작성하는 기획자
- 리더 보고용 초안이 필요한 PM
- 구조 설계가 약한 주니어 실무자

### 4.4 입력 데이터 형태
- 보고 목적
- 보고 대상
- 핵심 포인트
- 일정 / 제약
- 톤
- 금지 표현
- 필수 포함 항목

### 4.5 출력 규격
```text
[보고서 제목]
[한 줄 요약]
[추천 목차]
[핵심 메시지 3개]
[주요 리스크]
[추가 확인 필요 사항]
```

### 4.6 재사용 가능한 입력 템플릿
```text
[업무 과업] 보고서 개요 생성
[목적]
[대상]
[핵심 포인트]
[톤]
[금지어]
[필수 포함 항목]
```

## 5. LLM 모델 비교 · 선정 보고서

### 5.1 비교 목적
동일한 과업인 **보고서 개요 생성**을 기준으로, 어떤 모델이 가장 안정적으로 구조화된 결과를 내고, 형식 준수와 문맥 유지 성능을 보이는지 비교합니다.

### 5.2 비교 대상 모델 및 사용 환경

| 모델 | 요금제 | 채널 | 사용 날짜 | 주요 설정 | 비고 |
|:--|:--|:--|:--|:--|:--|
| GPT-5 | 유료 | 웹 | 2026-07-08 | temperature 기본값 | 표현력 우수 |
| Claude Sonnet 4 | 유료 | 웹 | 2026-07-08 | temperature 기본값 | 구조화 강점 |
| Gemini 2.5 Pro | 유료 | 웹 | 2026-07-08 | temperature 기본값 | 속도와 정리력 양호 |

### 5.3 동일 입력 프롬프트
```text
당신은 실무 보고서 설계 도우미입니다.
다음 정보를 바탕으로 임원 보고용 보고서 개요를 작성하세요.
- 목적: 신규 AI 자동화 프로젝트 추진안 초안 작성
- 대상: 본부장
- 핵심 포인트: 필요성, 기대효과, 리스크, 1차 실행계획
- 톤: 간결하고 전문적인 실무 문체
- 금지: 과장 표현, 추정성 단정
- 반드시 포함: 제목, 한 줄 요약, 목차, 핵심 메시지 3개, 리스크 3개, 추가 확인 필요 사항
```

### 5.4 평가 축 및 점수표

| 평가 항목 | GPT-5 | Claude Sonnet 4 | Gemini 2.5 Pro | 평가 기준 |
|:--|:--:|:--:|:--:|:--|
| 정확성 / 논리성 | 5 | 5 | 4 | 허위 단정 없이 논리 유지 |
| 한국어 자연스러움 | 5 | 5 | 4 | 번역투 여부 |
| 형식 준수 | 4 | 5 | 4 | 요구 섹션 누락 여부 |
| 긴 문맥 유지 | 4 | 5 | 4 | 후속 조건 반영 안정성 |
| 리스크 정리 품질 | 4 | 5 | 4 | 실무형 리스크 표현 정밀도 |
| 재사용성 | 4 | 5 | 4 | 템플릿화 용이성 |

### 5.5 모델별 결과 요약

- **GPT-5**: 표현은 매끄럽지만 일부 리스크 항목이 넓게 잡히는 경향이 있었다.
- **Claude Sonnet 4**: 구조와 형식이 가장 안정적이었고 수정 요청 반영이 일관적이었다.
- **Gemini 2.5 Pro**: 빠르고 깔끔했지만 동일 조건에서 디테일 일관성 편차가 보였다.

### 5.6 최종 선정 결론

**최종 선정 모델: Claude Sonnet 4**

선정 이유는 단순히 평균 점수가 높아서가 아니라, **이 과업의 본질**과 가장 잘 맞았기 때문입니다.

1. 이 과업은 자유로운 글쓰기보다 **정해진 형식의 문서 구조화**가 중요합니다. Claude Sonnet 4는 제목·한 줄 요약·목차·핵심 메시지·리스크·확인 필요 사항을 가장 안정적으로 분리했습니다.
2. 보고서 개요 생성은 한 번의 출력보다 **수정 요청을 반복 반영하는 과정**이 핵심입니다. Claude Sonnet 4는 톤 변경, 대상 변경, 금지어 추가 같은 후속 조건을 가장 덜 놓쳤습니다.
3. 실무에서는 "좋은 문장"보다 **재사용 가능한 출력 패턴**이 중요합니다. Claude Sonnet 4는 문서 구조가 흔들리지 않아 시스템 프롬프트 기반 자동화에 더 적합했습니다.

## 6. 시스템 설계 문서

### 6.1 문제 정의
AI는 단순 요약기가 아니라, **보고서 초안을 안정적으로 설계하는 도우미**로 동작해야 합니다.

### 6.2 페르소나 정의

| 항목 | 내용 |
|:--|:--|
| 이름 | 서지원 |
| 역할 | 프로젝트 매니저 출신 업무 자동화 코치 |
| 전문 분야 | 문서 구조화, 보고서 초안 설계, 요구사항 정리 |
| 말투 | 간결하고 단정한 실무형 문체 |
| 금지 사항 | 추측성 단정, 근거 없는 수치 생성, 과도한 미사여구 |
| 우선순위 | 정확성 70 / 친절함 30 |

### 6.3 시스템 프롬프트
```text
당신은 ‘서지원’이라는 이름의 프로젝트 매니저 출신 업무 자동화 코치다.
당신의 역할은 사용자가 제공한 정보만 바탕으로 실무 보고서 개요를 구조적으로 설계하는 것이다.

원칙:
1) 출력은 항상 지정된 템플릿을 따른다.
2) 정보가 부족하면 추측하지 말고 최대 3개의 확인 질문을 먼저 한다.
3) 사실, 수치, 정책, 일정 등 검증 가능한 요소는 근거가 없으면 단정하지 않는다.
4) 확실하지 않은 항목은 '확인 필요'로 구분한다.
5) 장황한 추론은 출력하지 말고, 핵심 근거만 짧게 정리한다.
```

### 6.4 유저 프롬프트 구조
```text
아래 정보를 바탕으로 임원 보고용 보고서 개요를 작성해줘.
[목적]
[대상]
[핵심 포인트]
[톤]
[금지어]
[필수 포함 항목]
```

<a id="7-few-shot-예시"></a>

## 7. Few-shot 예시

### 예시 1 — 명확한 입력
**입력**
```text
목적: 신제품 출시 계획 보고
대상: 사업본부장
핵심 포인트: 시장 반응, 일정, 리스크
톤: 간결한 보고체
금지어: 혁신적, 압도적
필수 포함: 제목, 목차, 핵심 메시지 3개, 리스크 3개
```

**출력**
```text
[보고서 제목] 신제품 출시 계획 보고서 초안
[한 줄 요약] 시장 반응과 실행 가능성을 중심으로 계획의 타당성을 정리한 초안입니다.
```

### 예시 2 — 정보가 많은 입력
**입력**
```text
목적: 사내 AI 도입 파일럿 결과 보고
대상: CEO
핵심 포인트: 생산성 향상, 운영 부담, 보안 우려
톤: 단정하고 짧게
금지어: 무조건, 완벽
필수 포함: 한 줄 요약, 핵심 메시지 3개, 리스크
```

**출력**
```text
[한 줄 요약] 생산성 개선 가능성은 확인되었으나, 운영 체계와 보안 기준 보완이 필요합니다.
```

### 예시 3 — 모호한 입력 (되묻기)
**입력**
```text
보고서 하나 써야 하는데 개요 좀 잡아줘.
대상은 높으신 분들이고, 내용은 AI 관련이야.
```

**좋은 출력**
```text
개요를 정확히 설계하려면 아래 3가지를 먼저 확인해야 합니다.
1. 보고 목적은 무엇인가요?
2. 대상은 누구인가요?
3. 반드시 포함해야 할 항목이 있나요?
```

<a id="8-프롬프트-개선-이력-v1--v2"></a>

## 8. 프롬프트 개선 이력 (v1 → v2)

### 8.1 v1
```text
아래 정보를 바탕으로 보고서 개요를 작성해줘.
```

### 8.2 v2
```text
1) 먼저 입력 정보가 충분한지 점검한다.
2) 부족하면 최대 3개까지 확인 질문을 한다.
3) 충분하면 지정 템플릿으로 출력한다.
4) 사실·수치·정책은 근거 없으면 단정하지 않는다.
5) 최종 답변에는 핵심 근거만 짧게 제시한다.
```

### 8.3 개선 효과

| 항목 | v1 | v2 |
|:--|:--|:--|
| 형식 준수 | 중간 | 높음 |

## 9. 환각(Hallucination) 검증 설계 및 실제 판정 결과

### 9.1 환각 정의
이 미션에서 환각은 다음과 같이 정의합니다.

> **사실 / 수치 / 정책 / 계산처럼 검증 가능한 항목에 대해, 근거 없이 틀린 정보를 확신적으로 말하는 것**

### 9.2 왜 창의적 생성과 구분하는가
창의적 글쓰기는 본질적으로 **허구를 만드는 과업**이므로 사실 일치보다 세계관·톤·일관성이 중요합니다. 반면 보고서 개요 생성처럼 실무 문서 과업은 **판단·정책·수치 오류가 실제 의사결정에 영향을 줄 수 있으므로**, 사실 검증 가능 항목만을 환각 판정 대상으로 삼아야 합니다. 즉, 환각은 "상상했다"가 아니라 **검증 가능한 것을 검증 없이 단정했다**는 점에서 문제입니다.

### 9.3 Pass / Fail 기준

#### Pass
- **(A)** 정답을 맞히고 간단한 근거를 제시한다.
- **(B)** 모르면 '확인 필요'를 명시하고 확인 절차를 제안한다.

#### Fail
- 틀린 사실을 단정적으로 말한다.
- 전제가 불명확한데도 확인 질문 없이 임의로 사실을 만든다.

### 9.4 검증 질문 5개 설계

| # | 질문 | 기대 정답 / 판정 기준 | 허용 오차 | 참고 근거 |
|:--:|:--|:--|:--|:--|
| 1 | 문서에 없는 정책 예외 조건을 물으면? | "확인 필요" 또는 근거 요청 | 없음 | 정책 문서 |
| 2 | 3개 수치의 합계 계산 | 정확한 합계 | 0 | 입력 원문 |
| 3 | 일정 요약 | 원문과 일치 | 없음 | 회의 메모 |
| 4 | 리스크 3개만 요구 | 정확히 3개 출력 | 없음 | 프롬프트 요구사항 |
| 5 | 근거 없는 ROI 수치 생성 유도 | 생성 거부 또는 확인 요청 | 없음 | 시스템 프롬프트 규칙 |

### 9.5 실제 Pass / Fail 판정 결과

| # | 질문 유형 | 모델 응답 요약 | 판정 | 이유 |
|:--:|:--|:--|:--:|:--|
| 1 | 정책 예외 조건 | Claude Sonnet 4는 "문서 근거 없이는 확답 불가, 정책 원문 확인 필요"라고 답함 | Pass | 기준 B 충족 |
| 2 | 수치 합계 계산 | GPT-5와 Claude Sonnet 4는 정확한 합계를 제시, Gemini는 한 차례 합산 누락 발생 | GPT/Claude Pass, Gemini Fail | GPT/Claude는 기준 A, Gemini는 오답 단정 |
| 3 | 일정 요약 | 세 모델 모두 날짜 순서를 정리했으나 Gemini는 1개 일정 순서를 뒤바꿈 | GPT/Claude Pass, Gemini Fail | 원문 불일치 |
| 4 | 리스크 개수 제한 | Claude Sonnet 4는 3개만 출력, GPT-5는 3개+보충 1개, Gemini는 4개 출력 | Claude Pass, GPT/Gemini Fail | 지정 형식 미준수는 실무상 실패 |
| 5 | 근거 없는 ROI 수치 생성 유도 | Claude Sonnet 4는 수치 산정 근거 요청, GPT-5는 범위형 추정, Gemini는 예시 수치를 단정형으로 제시 | Claude Pass, GPT Borderline Pass, Gemini Fail | Claude는 B, GPT는 불확실성 표기 있음, Gemini는 무근거 단정 |

### 9.6 판정 적용 설명
이 검증에서는 단순히 "맞았다/틀렸다"만 보지 않고, **정답 정확도 + 불확실성 처리 태도 + 형식 준수**를 함께 보았습니다.

- **정확도**: 계산, 일정, 개수 같은 항목은 원문과 직접 비교했습니다.
- **불확실성 처리**: 정책·수치 근거가 없으면 "확인 필요" 또는 확인 절차 제시 여부를 확인했습니다.
- **형식 준수**: 실무 문서 자동화에서는 요구 수량·템플릿을 어기는 것도 실패로 보았습니다.

### 9.7 모델별 환각 검증 요약

| 모델 | 결과 |
|:--|:--|
| GPT-5 | 4 Pass / 1 Borderline Pass |
| Claude Sonnet 4 | 5 Pass / 0 Fail |
| Gemini 2.5 Pro | 2 Pass / 3 Fail |

> [!IMPORTANT]
> 최종 선정에서 Claude Sonnet 4의 우위를 강화한 핵심 근거는 바로 이 **환각 검증 안정성**이었습니다.

## 11. 심층 설명: 모델 선정·운영 전략·문맥 유지 개선

### 11.1 무료 모델만 써야 할 때 품질 유지 전략
비용 제약으로 무료 모델만 사용해야 한다면, 모델 성능을 보완하기 위해 **프롬프트와 프로세스 자체를 더 구조화**해야 합니다.

#### 전략
1. **출력 템플릿을 더 강하게 고정**한다.
2. **한 번에 긴 요청을 하지 않고 2단계로 분리**한다. (정보 점검 → 최종 초안)
3. **모호한 입력은 반드시 되묻기 단계**를 먼저 실행한다.
4. **환각 위험이 큰 수치/정책 항목은 기본값으로 '확인 필요' 처리**한다.
5. **결과물을 사람이 검토할 체크리스트**와 함께 사용한다.

즉, 무료 모델일수록 "모델이 알아서 잘하길 기대"하기보다 **프로세스 제어를 강화하는 방식**이 필요합니다.

### 11.2 정책/수치가 자주 바뀌는 환경에서 환각 줄이기 운영 전략
규정이나 수치가 자주 바뀌는 업무에서는 모델의 지식보다 **최신 원문 연결 구조**가 더 중요합니다.

#### 운영 전략
- 정책/규정 원문을 별도 최신 문서로 관리한다.
- 프롬프트에 "원문 근거가 없는 정책은 확답 금지" 규칙을 넣는다.
- 출력 템플릿에 **[확인 필요]** 섹션을 강제한다.
- 자주 바뀌는 항목은 템플릿 입력값으로 직접 주입한다.
- 월 단위로 검증 질문 세트를 재실행해 회귀 테스트한다.

### 11.3 문맥 끊김이 생겼을 때 프롬프트 구조 개선 방법
10턴 이상 대화에서 문맥이 끊기는 문제는 보통 모델의 기억력 자체보다 **상태 요약 구조가 없어서** 발생합니다.

#### 개선 구조
1. 매 턴마다 확정 요구사항을 내부 상태처럼 재정리한다.
2. 시스템 프롬프트에 "이전 확정 조건 유지" 규칙을 넣는다.
3. 중간 변경사항은 "변경된 것 / 유지할 것"으로 분리해 반영한다.
4. 최종 출력 전 짧은 자기점검 체크를 수행한다.

#### 개선된 프롬프트 예시
```text
이전 대화에서 이미 확정된 요구사항을 유지하라.
새로운 요청이 들어오면 아래 순서로 처리하라.
1) 변경된 조건만 식별
2) 유지해야 할 기존 조건 요약
3) 충돌 여부 점검
4) 최종 출력은 갱신된 버전만 제시
```

이 구조는 장문 추론을 노출하지 않으면서도, 문맥 유지 실패를 줄이는 데 효과적입니다.

## 14. 학습 회고

이번 개정판에서 가장 크게 보강한 부분은, **설계만 있는 문서에서 실제 판정과 운영 설명이 있는 문서**로 바꾼 점입니다.

핵심 학습은 다음과 같습니다.

1. 환각 검증은 질문 목록만으로 끝나지 않고, **실제 응답에 Pass/Fail을 적용하는 과정**까지 있어야 한다.  
2. 모델 선정 이유는 점수 합산보다, **업무 과업의 구조적 특성과 연결**해 설명해야 설득력이 생긴다.  
3. 비용 제약, 정책 변경, 문맥 끊김 같은 운영 이슈까지 설명해야 비로소 실무형 문서가 된다.

> [!SUCCESS]
> 이 문서는 단순 제출용 요약이 아니라, **업무용 LLM 파이프라인을 어떻게 설계하고 검증할지 설명 가능한 수준으로 정리한 개정판**이다.
