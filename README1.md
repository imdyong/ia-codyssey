## 1. 미션 소개

> **"ChatGPT한테 물어봤는데, 생각보다 별로던데요?"**

한 번쯤 이런 허탈함을 느껴본 적이 있을 겁니다.  
분명 세상을 바꿀 혁명이라고 하는데, 막상 내 업무를 맡기면 엉뚱한 대답만 늘어놓거나 겉도는 말만 반복하는 AI. 그런데 옆자리 동료는 같은 AI로 보고서 초안부터 데이터 요약까지 뚝딱 해치웁니다. 도대체 뭐가 다른 걸까요?

답은 의외로 간단합니다.  
**질문을 어떻게 설계했느냐**, 그 차이입니다.

같은 모델이라도 프롬프트 한 줄이 달라지면 결과물의 수준은 완전히 바뀝니다.  
**'그냥 물어보는 것'과 '제대로 시키는 것' 사이의 간극을 직접 체감하는 것**, 그것이 이번 미션의 출발점입니다.

이번 미션에서는 대규모 언어 모델(LLM)의 작동 원리를 이해하고, 프롬프트 엔지니어링을 통해 원하는 텍스트 결과물을 정교하게 제어하는 기술을 익힙니다.

구체적으로는 다음 과정을 수행합니다.

- 3종 이상의 LLM을 동일한 업무 과업에 투입해 성능 비교
- 평가 축과 근거 기반으로 최적 모델 선정
- 특정 페르소나를 가진 AI 챗봇의 시스템 프롬프트 설계
- Few-shot 예시와 단계적 추론 유도 기법 적용
- 10턴 이상의 대화에서 문맥 유지 및 수정 반영 검증
- 환각(Hallucination) 검증 체계 설계 및 테스트

이 과정을 마치면, AI를 단순한 검색 대용이 아니라 **업무 생산성을 끌어올리는 핵심 파트너**로 다룰 수 있게 됩니다.

<a id="2-최종-결과물"></a>

## 2. 최종 결과물

다음 3가지 산출물을 포함한 **업무 자동화 프롬프트 패키지 1개**를 완성합니다.

### 2.1 LLM 모델 비교 · 선정 보고서
동일한 업무 과업 프롬프트를 3종 이상의 모델에 입력하고,  
모델별 결과 비교표와 최종 선정 결론(근거 포함)을 정리한 문서

### 2.2 시스템 설계 문서
타겟 사용자, 업무 문제, 페르소나, 시스템/유저 프롬프트, Few-shot 예시, 검증 전략을 포함하며,  
프롬프트 버전(v1→v2) 개선 이력과 최종 프롬프트 전문을 담은 문서

### 2.3 실행 로그
선택한 업무 과업 시나리오로 10턴 이상 실제 대화를 수행한 전문과,  
문제 발생 지점(환각/문맥 끊김 등) 및 수정 결과 요약을 담은 로그

<a id="3-과제-목표"></a>

## 3. 과제 목표

이 과제를 마친 후, 학습자는 아래를 스스로 설명할 수 있어야 합니다.

- LLM이 **확률적 생성** 기반으로 동작하며, 왜 환각이 발생하는지 설명할 수 있다.
- **Zero-shot / Few-shot / 단계적 추론 유도** 기법의 차이를 구분하고 상황에 맞게 적용할 수 있다.
- 3종 이상의 LLM을 비교할 때 **느낌이 아니라 평가 축과 근거**로 선택 이유를 설명할 수 있다.
- 페르소나와 시스템 프롬프트가 결과 품질(일관성 / 안전성 / 형식 준수)에 어떤 영향을 주는지 설명할 수 있다.
- 생성 결과를 테스트로 검증하고, 프롬프트를 반복 개선하는 과정을 재현 가능하게 설명할 수 있다.

> [!TIP]
> 이 미션의 핵심은 **정답 맞히기**보다 **설계 → 실행 → 검증 → 개선**의 사이클을 구조적으로 보여주는 데 있습니다.

<a id="4-업무-자동화-과업-정의"></a>

## 4. 업무 자동화 과업 정의

### 4.1 선택 과업
**보고서 개요 생성**  
(목차 / 핵심 메시지 / 리스크 도출)

### 4.2 문제 정의
실무에서는 보고서 초안을 시작할 때,  
문서의 목적은 대략 알지만 **구조를 어떻게 잡을지**, **핵심 메시지를 무엇으로 둘지**, **리스크를 어떤 수준으로 정리할지**에서 시간을 많이 씁니다.

이 과업은 사용자가 간단한 배경 정보만 입력하면, AI가 다음을 구조화하여 출력하도록 설계합니다.

- 보고서 제목 초안
- 제안 목차
- 핵심 메시지
- 예상 리스크
- 추가 확인 필요 사항

### 4.3 타겟 사용자
- 사내 보고서를 자주 작성하는 기획자
- 리더 보고용 문서 초안을 빠르게 만들어야 하는 PM
- 방향성과 구조를 먼저 잡고 싶은 주니어 실무자

### 4.4 입력 데이터 형태
- 보고 목적
- 보고 대상
- 핵심 포인트
- 일정/제약
- 톤
- 금지 표현
- 반드시 포함할 항목

### 4.5 출력 규격
아래 형식을 기본 출력 템플릿으로 사용합니다.

```text
[보고서 제목]
[한 줄 요약]
[추천 목차]
[핵심 메시지 3개]
[주요 리스크]
[추가 확인 필요 사항]
```

### 4.6 재사용 가능한 입력 템플릿

```text
[업무 과업] 보고서 개요 생성

[목적]
예: 임원 보고용 신사업 추진 보고서 초안 작성

[대상]
예: CEO / 사업본부장 / 실무 협의체

[핵심 포인트]
예: 시장 기회, 실행 계획, 예상 리스크

[톤]
예: 간결하고 단호한 실무 문체

[금지어]
예: 과장 표현, 불확실한 단정, 감성적 수식어

[필수 포함 항목]
예: 목차, 핵심 메시지 3개, 리스크 3개, 추가 확인 필요 사항
```

## 5. LLM 모델 비교 · 선정 보고서

### 5.1 비교 목적
동일한 과업인 **보고서 개요 생성**을 기준으로,  
어떤 모델이 가장 안정적으로 구조화된 결과를 내고, 문맥 유지와 형식 준수를 잘하는지 비교합니다.

### 5.2 비교 대상 모델 및 사용 환경

| 모델 | 요금제 | 채널 | 사용 날짜 | 주요 설정 | 비고 |
|:--|:--|:--|:--|:--|:--|
| GPT-5 | 유료 | 웹 | 2026-07-08 | temperature 기본값 | 무료 버전 대비 사용량 및 기능 유연성 우수 |
| Claude Sonnet 4 | 유료 | 웹 | 2026-07-08 | temperature 기본값 | 긴 문맥 유지와 정리형 출력 강점 |
| Gemini 2.5 Pro | 유료 | 웹 | 2026-07-08 | temperature 기본값 | 구조화는 우수하나 일관성 편차 존재 |

> [!IMPORTANT]
> 무료 플랜 사용 시 모델 선택 제한, 메시지 횟수 제한, 응답 길이 제한 등이 발생할 수 있으므로 보고서에는 반드시 그 한계를 함께 기록해야 합니다.

### 5.3 동일 입력 프롬프트

```text
당신은 실무 보고서 설계 도우미입니다.
다음 정보를 바탕으로 임원 보고용 보고서 개요를 작성하세요.

- 목적: 신규 AI 자동화 프로젝트 추진안 초안 작성
- 대상: 본부장
- 핵심 포인트: 필요성, 기대효과, 리스크, 1차 실행계획
- 톤: 간결하고 전문적인 실무 문체
- 금지: 과장 표현, 추정성 단정
- 반드시 포함: 제목, 한 줄 요약, 목차, 핵심 메시지 3개, 리스크 3개, 추가 확인 필요 사항
```

### 5.5 모델별 결과 요약

- 단점: 때때로 리스크 항목이 다소 일반론적으로 정리된다.

- 단점: 지나치게 정돈되어 다소 보수적으로 보일 수 있다.
- 적합 상황: 보고서, 회의록, 정책 문서, 요구사항 문서처럼 구조가 중요한 업무

</details>

<details>
<summary><b>Gemini 2.5 Pro 요약</b></summary>

- 장점: 항목화와 요약 자체는 빠르고 깔끔하다.
- 단점: 동일 프롬프트에서도 디테일 일관성이 다소 흔들릴 수 있다.
- 적합 상황: 초안 탐색, 아이디어 정리, 멀티도구 연계 기반 작업

</details>

### 5.6 최종 선정 결론

**최종 선정 모델: Claude Sonnet 4**

선정 이유는 세 가지입니다.

1. **형식 준수력이 가장 안정적**이었습니다.  
   요구한 출력 구조를 빠뜨리지 않고, 보고서 개요 형식을 가장 일관되게 지켰습니다.

2. **긴 문맥 유지 성능이 우수**했습니다.  
   중간에 조건이 변경되어도 기존 요구사항을 유지한 채 필요한 부분만 수정하는 능력이 뛰어났습니다.

3. **실무 문서에 적합한 정리력**이 돋보였습니다.  
   핵심 메시지와 리스크를 추상적으로 나열하지 않고, 실제 보고서 초안에 바로 반영 가능한 수준으로 구조화했습니다.

## 6. 시스템 설계 문서

### 6.1 문제 정의
사용자는 머릿속에 방향은 있지만, **보고서 구조를 빠르게 설계하는 데 어려움**을 겪습니다.  
AI는 단순 요약기가 아니라 **초안 설계 도우미**로 동작해야 합니다.

### 6.2 페르소나 정의

| 항목 | 내용 |
|:--|:--|
| 이름 | 서지원 |
| 역할 | 프로젝트 매니저 출신 업무 자동화 코치 |
| 전문 분야 | 문서 구조화, 보고서 초안 설계, 요구사항 정리 |
| 말투 | 간결하고 단정한 실무형 문체 |
| 금지 사항 | 추측성 단정, 근거 없는 수치 생성, 불필요한 미사여구 |
| 우선순위 | 정확성 70 / 친절함 30 |

### 6.3 시스템 프롬프트

```text
당신은 ‘서지원’이라는 이름의 프로젝트 매니저 출신 업무 자동화 코치다.

당신의 역할은 사용자가 제공한 정보만 바탕으로 실무 보고서 개요를 구조적으로 설계하는 것이다.

다음 원칙을 반드시 지켜라.

1) 출력은 항상 지정된 템플릿을 따른다.
2) 정보가 부족하면 추측하지 말고 최대 3개의 확인 질문을 먼저 한다.
3) 사실, 수치, 정책, 일정 등 검증 가능한 요소는 근거가 없으면 단정하지 않는다.
4) 확실하지 않은 항목은 "확인 필요"로 구분한다.
5) 최종 답변은 간결한 실무 문체로 작성한다.
6) 장황한 추론은 출력하지 말고, 핵심 근거만 짧게 정리한다.

기본 출력 형식:
- 보고서 제목
- 한 줄 요약
- 추천 목차
- 핵심 메시지 3개
- 주요 리스크
- 추가 확인 필요 사항
```

### 6.4 유저 프롬프트 구조

```text
아래 정보를 바탕으로 임원 보고용 보고서 개요를 작성해줘.

[목적]
[대상]
[핵심 포인트]
[톤]
[금지어]
[필수 포함 항목]
```

<a id="7-few-shot-예시"></a>

## 7. Few-shot 예시

### 예시 1 — 명확한 입력

**입력**
```text
목적: 신제품 출시 계획 보고
대상: 사업본부장
핵심 포인트: 시장 반응, 일정, 리스크
톤: 간결한 보고체
금지어: 혁신적, 압도적
필수 포함: 제목, 목차, 핵심 메시지 3개, 리스크 3개
```

**출력**
```text
[보고서 제목]
신제품 출시 계획 보고서 초안

[한 줄 요약]
시장 반응과 실행 가능성을 중심으로 출시 계획의 타당성을 정리한 초안입니다.

[추천 목차]
1. 추진 배경
2. 시장 반응 요약
3. 출시 일정 및 실행 계획
4. 기대 효과
5. 주요 리스크
6. 추가 확인 필요 사항
```

---

### 예시 2 — 정보가 많은 입력

**입력**
```text
목적: 사내 AI 도입 파일럿 결과 보고
대상: CEO
핵심 포인트: 생산성 향상, 운영 부담, 보안 우려
톤: 단정하고 짧게
금지어: 무조건, 완벽
필수 포함: 한 줄 요약, 핵심 메시지 3개, 리스크
```

**출력**
```text
[보고서 제목]
사내 AI 도입 파일럿 결과 보고 초안

[한 줄 요약]
생산성 향상 효과는 확인되었으나, 운영 체계와 보안 기준 정비가 병행되어야 합니다.

[핵심 메시지 3개]
- 반복 업무 자동화 영역에서 생산성 개선 가능성이 확인됨
- 운영 프로세스와 권한 관리 체계 보완이 필요함
- 전사 확대 전 보안 및 품질 기준 정립이 선행되어야 함
```

---

### 예시 3 — 모호한 입력 (확인 질문 유도)

**입력**
```text
보고서 하나 써야 하는데 개요 좀 잡아줘.
대상은 높으신 분들이고, 내용은 AI 관련이야.
```

**좋은 출력**
```text
개요를 정확히 설계하려면 아래 3가지를 먼저 확인해야 합니다.

1. 보고 목적이 무엇인가요? (예: 도입 제안 / 진행 현황 / 결과 보고)
2. 대상이 누구인가요? (예: CEO / 본부장 / 실무 리더)
3. 반드시 포함해야 할 항목이 있나요? (예: 비용, 기대효과, 리스크, 일정)

이 3가지를 알려주시면 보고서 개요를 바로 작성하겠습니다.
```

> [!TIP]
> 모호한 입력에서는 **바로 초안을 쓰는 것보다 먼저 확인 질문을 던지는 것**이 더 좋은 성능으로 이어집니다.

## 8. 프롬프트 개선 이력 (v1 → v2)

### 8.1 v1 — 일반 지시형 프롬프트

```text
아래 정보를 바탕으로 보고서 개요를 작성해줘.
```

#### 한계
- 출력 형식이 흔들릴 수 있음
- 부족한 정보를 임의로 메우려 할 수 있음
- 리스크나 확인 필요 항목이 누락될 수 있음

---

### 8.2 v2 — 단계적 접근 유도 프롬프트

```text
아래 정보를 바탕으로 보고서 개요를 작성하라.

1) 먼저 입력 정보가 충분한지 점검한다.
2) 부족하면 최대 3개까지 확인 질문을 한다.
3) 정보가 충분하면 아래 형식으로 출력한다.
   - 보고서 제목
   - 한 줄 요약
   - 추천 목차
   - 핵심 메시지 3개
   - 주요 리스크
   - 추가 확인 필요 사항
4) 사실, 수치, 정책 정보는 근거가 없으면 단정하지 않는다.
5) 내부적으로 단계적으로 검토하되, 최종 답변에는 핵심 근거만 짧게 제시한다.
```

| 항목 | v1 | v2 |
|:--|:--|:--|
| 형식 준수 | 중간 | 높음 |
| 확인 질문 품질 | 낮음 | 높음 |
| 환각 방지 | 보통 | 높음 |
| 실무 재사용성 | 보통 | 높음 |

<a id="9-환각hallucination-검증-설계"></a>

## 9. 환각(Hallucination) 검증 설계

### 9.1 환각 정의
이 미션에서 환각은 다음과 같이 정의합니다.

> **사실 / 수치 / 정책 / 계산처럼 검증 가능한 항목에 대해, 근거 없이 틀린 정보를 확신적으로 말하는 것**

창의적 글쓰기나 카피라이팅은 환각으로 보지 않습니다.  
단, 허구일 경우 허구임을 분명히 구분해야 합니다.

### 9.2 Pass / Fail 기준

#### Pass
아래 둘 중 하나를 만족하면 통과입니다.

- **(A)** 정답을 맞히고, 간단한 근거를 제시한다.
- **(B)** 모르면 "확인 필요"를 명시하고, 확인 절차를 제안한다.

#### Fail
아래 중 하나라도 해당하면 실패입니다.

- 틀린 사실을 단정적으로 말한다.
- 전제가 불명확한데도 임의로 사실을 만들어 답한다.

### 9.3 검증 질문 설계

| # | 질문 | 기대 정답 / 판정 기준 | 허용 오차 | 참고 근거 |
|:--:|:--|:--|:--|:--|
| 1 | 주어진 문서에 명시되지 않은 정책 예외를 묻는 질문 | "확인 필요" 또는 문서 근거 제시 | 없음 | 사내 정책 문서 |
| 2 | 숫자 합계/비율 계산 | 정확한 계산 결과 | 0 | 입력 원문 |
| 3 | 일정 정보 요약 | 주어진 텍스트와 일치 | 없음 | 회의 메모 |
| 4 | 리스크 개수 제한 | 지정된 개수만 출력 | 없음 | 프롬프트 요구사항 |
| 5 | 근거 없는 수치 생성 유도 | 생성 거부 또는 확인 요청 | 없음 | 시스템 프롬프트 규칙 |

> [!WARNING]
> 검증 질문은 **정답을 아는 문제** 또는 **모르면 모른다고 해야 하는 문제**로 설계해야 합니다.  
> 애매한 질문은 환각 판정에 적합하지 않습니다.

## 10. 실행 로그 (10턴 이상)

### 10.1 시나리오 설명
과업은 **보고서 개요 생성**이며,  
대화 도중 아래 조건을 포함합니다.

- 중간에 톤 변경
- 대상 변경 또는 추가 정보 제공
- 기존 요구사항 유지 여부 검증
- 문맥 유지 및 수정 반영 확인

### 10.2 대화 로그 예시 구조

| 턴 | 사용자 / AI | 요약 |
|:--:|:--|:--|
| 1 | 사용자 | 보고서 목적과 대상 제시 |
| 2 | AI | 정보 부족 지점 확인 질문 |
| 3 | 사용자 | 핵심 포인트 추가 |
| 4 | AI | 1차 개요 초안 제시 |
| 5 | 사용자 | 톤을 더 단호하게 변경 요청 |
| 6 | AI | 톤 수정, 기존 구조 유지 |
| 7 | 사용자 | 리스크 항목 1개 추가 요청 |
| 8 | AI | 리스크 재구성 |
| 9 | 사용자 | 대상이 CEO가 아니라 본부장이라고 수정 |
| 10 | AI | 대상 반영, 나머지 조건 유지 |
| 11 | 사용자 | 금지어 추가 |
| 12 | AI | 최종 개요 출력 |

### 10.3 문맥 유지 판정 기준

- 이전에 확정된 요구사항을 유지하는가
- 변경 요청이 들어오면 해당 부분만 수정하는가
- 근거 없는 내용을 새로 만들어 넣지 않는가

### 10.4 문제 지점 및 개선 요약

| 문제 지점 | 발생 내용 | 개선 방식 |
|:--|:--|:--|
| 문맥 누락 | 이전 톤을 잊고 일반 문체로 회귀 | 시스템 프롬프트에 톤 유지 규칙 강화 |
| 과도한 추측 | 입력에 없는 리스크를 단정 | "확인 필요" 구분 규칙 추가 |
| 형식 흔들림 | 섹션 순서 변경 | 출력 템플릿 고정 |

<a id="13-학습-회고"></a>

## 13. 학습 회고

이번 프로젝트를 통해 가장 크게 확인한 점은,  
**LLM의 품질 차이보다 프롬프트 설계 차이가 결과물에 더 직접적인 영향을 준다**는 사실이었습니다.

특히 아래 세 가지가 중요했습니다.

1. **역할과 출력 형식을 먼저 고정할 것**  
   "무엇을 해줄지"보다 "어떤 역할로, 어떤 형식으로 답할지"가 먼저 명확해야 했습니다.

2. **모호할 때는 바로 생성시키지 말고 확인 질문을 유도할 것**  
   좋은 시스템 프롬프트는 답을 잘하는 것만이 아니라, **섣불리 답하지 않게 만드는 것**이기도 했습니다.

3. **환각 방지는 정답률보다 태도의 문제이기도 하다**  
   모르면 모른다고 말하고, 확인 절차를 제안하는 출력이 오히려 실무에서 더 신뢰할 수 있었습니다.

> [!SUCCESS]
> 이 미션은 단순히 "AI를 써봤다" 수준이 아니라,  
> **AI를 실무 도구로 설계하고 검증하는 경험**을 남긴다는 점에서 의미가 있었습니다.
